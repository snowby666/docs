## [2026-06-11] - OpenAI models: monthly token limits

OpenAI models now have a **monthly token budget** per subscription plan, mirroring the existing Anthropic (Claude) monthly limit. The budget caps combined input + output tokens and resets at **21:00 UTC on the 1st of each month**. There is no separate daily request cap for OpenAI models — only the standard per-minute rate limit and this monthly token budget apply.

| Subscription Plan | Monthly Token Limit |
|---|---|
| **Free** | 100K tokens |
| **Starter** | 10M tokens |
| **Plus** | 25M tokens |
| **Core** | 50M tokens |
| **Pro** | 100M tokens |
| **Business** | 200M tokens |
| **Enterprise** | 300M tokens |

- **Applies to:** all OpenAI models (e.g. `gpt-4o`, `gpt-4.1`, `o3`, `o4-mini`) across `/chat/completions`, `/messages`, and `/responses`.
- **Exemptions:** Pay-as-you-go users with active permanent credits are billed per use and bypass the budget; admin keys are unlimited.
- **Exceeding the budget** returns a `429` with a message indicating the limit and reset date, independent of your per-minute rate limit.
- See [Rate Limits](/getting-started/rate-limits#openai-models-monthly-token-limits) for full details.

## [2026-05-23] - DDoS hardening: dynamic IDs move from URL path to request body

A long path-traversal-append DDoS attack (`GET /v1/models/<random>/<random>`, `GET /v1/stats/models/<random>/<random>`, etc.) made it impossible to safely whitelist dynamic-path GETs at the Cloudflare edge without a regex-capable plan. To fix this without an enterprise plan upgrade, every endpoint that previously took an `{order_id}`, `{proxy_key}`, or `{model_id}` in the URL now has a static-URL POST sibling that takes the same identifier in the JSON body. The URL is therefore an exact-match at the edge and the appending-random-segments attack pattern becomes meaningless.

### Added &mdash; new POST endpoints (identifier in JSON body)

All new endpoints accept a single field in the body and return the same shape as their original GET counterpart. They all inherit the original auth requirements (see "Auth model" below).

| New endpoint | Body | Replaces (still works) |
|---|---|---|
| `POST /auth/paypal/transaction` + `/v1/...` | `{"order_id": "..."}` | `GET /auth/paypal/transaction/{order_id}` |
| `POST /auth/paypal/invoice` + `/v1/...` | `{"order_id": "..."}` | `GET /auth/paypal/invoice/{order_id}` |
| `POST /auth/paypal/invoice/send` + `/v1/...` | `{"order_id": "..."}` | `POST /auth/paypal/invoice/{order_id}/send` |
| `POST /auth/payment/crypto/transaction` + `/v1/...` | `{"order_id": "..."}` | `GET /auth/payment/crypto/transaction/{order_id}` |
| `POST /auth/proxy/lookup` + `/v1/...` | `{"proxy_key": "..."}` | `GET /auth/proxy/{proxy_key}` |
| `POST /auth/proxy/update` + `/v1/...` | `{"proxy_key": "...", "name": "...", "expires_at": ..., "allocated_ammount": ..., "model_whitelist": [...], "ip_whitelist": [...]}` | `POST /auth/proxy/update/{proxy_key}` |
| `POST /auth/proxy/toggle` + `/v1/...` | `{"proxy_key": "..."}` | `POST /auth/proxy/toggle/{proxy_key}` |
| `POST /auth/proxy/remove` + `/v1/...` | `{"proxy_key": "..."}` | `DELETE /auth/proxy/delete/{proxy_key}` |
| `POST /stats/model` + `/v1/...` | `{"model_id": "..."}` | `GET /stats/models/{model_id}` |

All body fields are Pydantic-validated with `min_length=1, max_length=256` so attackers cannot push huge payloads.

### Auth model

- **Required (API key or JWT):** `/auth/paypal/transaction`, `/auth/paypal/invoice`, `/auth/paypal/invoice/send`, `/auth/payment/crypto/transaction`.
- **Required (API key only) + ownership check:** `/auth/proxy/lookup`, `/auth/proxy/update`, `/auth/proxy/toggle`, `/auth/proxy/remove` &mdash; each returns 404 if the proxy key doesn't exist and 403 if the caller is not its owner.
- **Public (no auth):** `/stats/model` &mdash; per-model statistics are public information consumed by the anonymous rankings page.

### Security fixes (pre-existing bugs found during this audit)

- **`DELETE /auth/proxy/delete/{proxy_key}`** previously had **no auth dependency and no ownership check** &mdash; any unauthenticated caller who knew or guessed a proxy key value could delete it. It now requires an API key and verifies that the caller owns the key (`404` if not found, `403` if not owned).
- **`POST /auth/proxy/update/{proxy_key}`** had an auth dependency but **no ownership check** &mdash; any authenticated user could overwrite anyone else's proxy key (name, allocated amount, model whitelist, IP whitelist) by knowing its value. It now verifies ownership (`404` / `403`).
- **`GET /auth/proxy/{proxy_key}`** previously had no ownership check, so anyone who knew or guessed a proxy key could read its metadata. It now requires an API key and verifies ownership.

### Changed &mdash; existing GETs are now authenticated

These GETs were previously anonymous and are now behind authentication so that any leftover path-traversal hits return `401` cheaply (FastAPI rejects them at the dependency stage, before any DB or cache work):

- `GET /models/{model}` and `GET /v1/models/{model}` &mdash; require API key or JWT.
- `GET /nsfw/models/{model}` &mdash; require API key or JWT.
- `GET /stats/models/{model_id}` and `GET /v1/stats/models/{model_id}` &mdash; require API key or JWT. (The public landing page now uses the new `POST /v1/stats/model` instead.)
- `GET /auth/proxy/{proxy_key}` and `GET /v1/auth/proxy/{proxy_key}` &mdash; require API key + ownership check.

`GET /models` and `GET /v1/models` (list-all) remain public so the playground model picker keeps working for anonymous visitors.

### Frontend migration

- **`playground.electronhub.ai/console`** &mdash; all five payment lookup `fetch` calls were converted to the new POST endpoints.
- **`electronhub.ai/rankings`** &mdash; the model detail modal now POSTs to `/v1/stats/model` with `{ model_id }` instead of `GET /v1/stats/models/<id>`.

### Migration

- **No action required** for clients of the public OpenAI-style endpoints (`/v1/chat/completions`, `/v1/embeddings`, etc.) &mdash; they are completely unaffected.
- **Clients that previously hit `GET /auth/proxy/{proxy_key}` anonymously** must now send a Bearer token. Recommended: migrate to `POST /auth/proxy/lookup` with `{"proxy_key": "..."}` in the body.
- **Clients polling `GET /auth/paypal/transaction/{order_id}` / `GET /auth/payment/crypto/transaction/{order_id}`** should switch to the new POST variants. The old GET routes continue to work (still authenticated as before) but will be removed in a future release.

## [2026-05-15] - Separate reasoning streams for chat completions

### Added
- **Separate `reasoning` field** on `/v1/chat/completions`. Reasoning models now return their chain-of-thought in a dedicated `reasoning` field instead of inline `<think>...</think>` tags inside `content`. Streaming deltas use `delta.reasoning`.
- **`POST /v1legacy/chat/completions`** &mdash; identical request/response shape, but reasoning uses the legacy field name `reasoning_content` (for older DeepSeek-style clients).
- **`POST /v1thinking/chat/completions`** &mdash; merges reasoning and answer into the normal `content` stream wrapped in `<think>...</think>` (the pre-migration behavior).
- New request fields on chat completions:
  - `reasoning` object (`{ effort, exclude, delta_field }`)
  - `reasoning_delta_field` (`"reasoning"` or `"reasoning_content"`)
  - `reasoning_content_compat` (boolean)
  - `:reasoning-exclude` model suffix to strip reasoning for a single call
- `reasoning_effort` now also accepts `"none"`, `"minimal"`, and `"xhigh"` in addition to `low`/`medium`/`high`.
- Playground (`playground.electronhub.ai`) consumes the new separate `reasoning` field and renders it in the existing collapsible thinking container.

### Changed
- `reasoning_effort` is **silently ignored** for models that don't support it instead of returning HTTP 400. This lets clients share a single request shape across providers.

### Migration
- **No action required** for clients that ignore reasoning fields &mdash; `content` keeps containing the final answer.
- Clients that previously parsed `<think>...</think>` from `content` should either:
  1. Switch to consuming the new `reasoning` (or `reasoning_content`) field, or
  2. Point at `/v1thinking/chat/completions` to keep the old inline behavior.

## API Versioning

We use semantic versioning for our API:
- **Major version** (X.0.0): Breaking changes that require code updates
- **Minor version** (X.Y.0): New features that are backward compatible  
- **Patch version** (X.Y.Z): Bug fixes and improvements

### Version Support
- **Current version**: v1.2.0 (fully supported)
- **Previous version**: v1.1.0 (security updates only)
- **Legacy versions**: v1.0.x (deprecated, sunset in 6 months)

## Breaking Changes Policy

We avoid breaking changes whenever possible. When necessary:
- 90-day advance notice via email and documentation
- Migration guides provided
- Legacy endpoint support during transition period
- Developer support for migration assistance
